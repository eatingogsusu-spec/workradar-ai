# WorkRadar AI

운영 문서를 올리면 할 일과 이슈, 주간·월간 보고서, 인수인계 문서를 자동으로 뽑아주는 웹 애플리케이션입니다. 자동차 부품 회사의 회의록·구매 이메일·품질 클레임 같은 문서를 다룬다고 가정하고 만들었습니다.

5인 학생 개발팀 프로젝트로 시작했고, 저는 AI/RAG 파이프라인을 맡았습니다. 이 저장소는 발표 이후 제가 개인 포트폴리오로 이어서 작업한 버전으로, 클라우드 의존성을 걷어내고 전부 로컬에서 돌아가도록 다시 만든 결과물입니다.

## 무엇을 하는가

문서를 업로드하면 다음을 자동으로 만들어냅니다.

- **할 일·이슈 추출** — 회의록이나 이메일에서 "누가 언제까지 무엇을" 형태의 실행 항목과, 리스크가 될 만한 이슈를 뽑아냅니다. 담당자와 마감일은 원문에 근거가 있을 때만 채웁니다.
- **AI 어시스턴트** — "KET 공급처 납기 지연 상황 알려줘" 같은 질문에 등록된 문서를 근거로 답합니다. 근거가 없으면 지어내지 않고 "찾지 못했다"고 답합니다.
- **보고서 생성** — 축적된 할 일·이슈를 바탕으로 주간·월간 운영 보고서를 8개 섹션으로 작성합니다.
- **인수인계 문서** — 담당자가 바뀔 때, 진행 중인 이슈와 거기서 파생된 할 일을 인과관계로 묶어 인수인계 문서를 만듭니다.

## 기술 스택

- **백엔드**: FastAPI, PostgreSQL + pgvector
- **프론트엔드**: React, 정적 JS
- **LLM**: Ollama (EXAONE 3.5 — 한국어 특화 모델)
- **임베딩**: nomic-embed-text (768차원)
- **인프라**: Docker

## 아키텍처

```
문서 업로드
   → 파싱 / 청킹
   → 임베딩(nomic) → pgvector 저장
   → LLM(EXAONE)으로 요약·추출
   → 할 일 / 이슈 / 보고서 / 인수인계 생성

질의(RAG)
   → 질문 임베딩 → pgvector 유사도 검색
   → 근거 문서 + 구조화 데이터를 컨텍스트로 구성
   → LLM이 근거 기반 답변 생성
```

## 이 프로젝트에서 실제로 부딪힌 문제들

기능을 나열하는 것보다, 만드는 과정에서 무엇이 막혔고 어떻게 풀었는지가 더 솔직한 기록이라고 생각합니다. 대부분은 "돌아가는 것처럼 보이는데 사실은 아닌" 문제들이었습니다.

### 클라우드에서 완전 로컬로 전환

원래는 Azure OpenAI와 Azure PostgreSQL을 쓰는 구조였습니다. 발표가 끝나고 구독이 만료되면서, 포트폴리오로 계속 쓰려면 클라우드 없이 돌아가게 만들어야 했습니다.

- LLM을 Azure OpenAI에서 **Ollama(EXAONE 3.5)** 로 교체
- 임베딩을 Azure 임베딩(1536차원)에서 **nomic-embed-text(768차원)** 로 교체
- 벡터 저장소를 Azure PostgreSQL에서 **로컬 Docker의 pgvector** 로 교체

코드 곳곳에 `AI_PROVIDER != "azure"`처럼 클라우드를 전제한 가드가 박혀 있어서, 이걸 전부 provider 중립적으로 바꾸는 게 생각보다 일이었습니다. 하나라도 놓치면 그 기능만 조용히 폴백(fallback) 처리돼서, 겉으로는 멀쩡해 보이는데 실제로는 AI가 안 돌아가는 상태가 됐습니다.

### 임베딩이 안 되던 세 겹의 원인

로컬로 옮긴 뒤 검색이 계속 비어서 나왔습니다. 원인이 하나가 아니라 세 개가 겹쳐 있었습니다.

1. **청크가 아예 없었음** — 시드 데이터가 문서 원본만 넣고 청크(chunk)는 만들지 않았습니다. 문서 텍스트는 DB가 아니라 디스크에 있어서, 임베딩할 대상 자체가 없는 상태였습니다. 문서를 다시 파싱해서 청킹하는 스크립트를 따로 만들었습니다.
2. **벡터 차원 불일치** — DB 컬럼은 `vector(1536)`(Azure 기준)인데 nomic은 768차원을 내놓습니다. 차원이 안 맞으면 삽입 자체가 실패합니다. 컬럼을 768로 마이그레이션하고, 스키마 파일도 함께 고쳐서 DB를 새로 만들어도 같은 문제가 재발하지 않게 했습니다.
3. **모델 이름이 잘못 기록되던 버그** — 로컬 768 벡터가 Azure 모델 이름으로 저장되고 있어서, 나중에 둘을 구분할 수 없게 될 뻔했습니다.

그리고 데이터가 88건밖에 안 되는데 ivfflat 인덱스가 걸려 있어서, 상위 3개를 요청해도 1개만 반환되는 문제가 있었습니다. pgvector가 "데이터가 적으면 recall이 떨어진다"고 경고까지 띄웠는데, 소규모에서는 인덱스 없이 전체 스캔이 오히려 정확하고 빨라서 인덱스를 제거했습니다.

### AI가 헛소리를 하던 진짜 이유

가장 오래 헤맨 문제입니다. 어시스턴트에게 특정 품번을 물으면, 그 품번을 한 번도 언급하지 않고 엉뚱한 문서를 요약해서 답했습니다. 처음엔 검색이 부실한 줄 알고 검색 쪽을 파고들었는데, 실측해보니 검색은 이미 잘 되고 있었습니다(상위 5개 정답률 5/5).

진짜 원인은 **Ollama의 기본 컨텍스트 창(num_ctx)이 2048토큰**이라는 데 있었습니다. 근거로 넘긴 데이터가 8000토큰이 넘는데, Ollama가 에러도 경고도 없이 앞부분을 잘라버리고 있었습니다. 프롬프트가 `[지시문 → 데이터]` 순서라, 잘려나간 게 정확히 지시문이었습니다. 모델은 "어떻게 답하라"는 지시를 아예 못 받은 채 데이터 꼬리만 보고 답하고 있었던 겁니다.

컨텍스트 창을 키우자 이 문제 하나로 묶여 있던 세 가지 증상(보고서 생성 실패, 어시스턴트 헛소리, 답변 잘림)이 한꺼번에 해결됐습니다. 원인을 코드나 프롬프트가 아니라 런타임 설정에서 찾아야 했던 경우였습니다.

### 로컬 모델의 환각을 후처리로 걸러내기

로컬 모델은 클라우드 모델보다 지시를 덜 정확하게 따릅니다. 근거에 없는 할 일이나 일정을 만들어내는 일이 종종 있었습니다. 프롬프트로 "지어내지 마라"를 강조해도 완전히 막지는 못했습니다.

그래서 프롬프트에만 의존하지 않고, **생성된 답변이 인용한 항목을 실제 데이터와 대조해서 존재하지 않는 것은 걸러내는** 후처리를 넣었습니다. 모델이 없는 할 일을 지어내도, 사용자에게 나가기 전에 검증 단계에서 제거됩니다. 인수인계에서도 같은 방식으로, 할 일과 이슈의 실제 연결 관계(`linked_issue_id`)를 근거로 넘겨서 모델이 관계를 추측하지 않게 했습니다.

### 문서 추출 타임아웃

큰 문서를 올리면 "분석 시간 초과"로 실패했습니다. 그런데 백엔드는 정상적으로 처리를 끝내고 있었고, **프론트엔드가 먼저 포기**하는 게 원인이었습니다. 폴링 예산이 약 56초인데 실제 추출이 그걸 살짝 넘겼습니다.

프론트 대기 시간을 넉넉하게 늘리는 동시에, 추출 자체도 빠르게 만들었습니다. 파이프라인이 같은 문서를 요약과 추출에서 두 번 통째로 LLM에 넣고 있었는데, 두 프롬프트를 "문서 먼저" 순서로 통일해서 Ollama의 프롬프트 캐시가 재사용되도록 했습니다. 지시문은 한 글자도 바꾸지 않고 순서만 바꿔서, 추출 시간을 33% 줄였습니다.

### GPU 없는 환경에서의 속도 최적화

응답이 느려서 원인을 실측으로 분석했습니다. 프롬프트 처리 비용이 토큰당 약 2.5ms로 토큰 수에 정확히 선형이었고, 시간의 대부분이 답을 생성하기 전 "프롬프트를 읽는" 단계에서 소모되고 있었습니다. VRAM 한계로 모델과 KV 캐시가 GPU에 다 올라가지 못하는 게 근본 원인이었습니다.

컨텍스트 창 크기나 출력 토큰 상한을 조절하는 건 속도에 효과가 없었고(실측으로 확인), **근거로 넘기는 항목 수를 관련도 상위로 줄이는 것**이 유효했습니다. 검색은 이미 관련도 순으로 정렬돼 있어서, 잘려나가는 건 가장 덜 관련되고 가장 비싼 꼬리 부분이었습니다. 이걸로 어시스턴트 응답 시간을 약 40% 줄였습니다.

## 배운 점

이 프로젝트에서 가장 많이 한 일은 "돌아가는 것처럼 보이는데 사실은 아닌" 상태를 찾아내는 것이었습니다. 폴백으로 조용히 넘어가거나, 에러 없이 잘려나가거나, 그럴듯한 헛소리로 채워지는 문제들은 로그만 봐서는 안 보이고, 실제로 측정하고 데이터와 대조해야 드러났습니다.

특히 "검색이 문제일 것"이라는 처음 가설이 실측 결과 틀렸던 경험(진짜 원인은 컨텍스트 창이었음)을 통해, 추측으로 방향을 잡기보다 먼저 재현하고 측정하는 습관의 중요성을 배웠습니다.

## 실행 방법

로컬 실행에 필요한 전체 셋업 순서(모델 다운로드, DB 스키마·시드, 임베딩 생성 등)는 별도 셋업 가이드에 정리되어 있습니다. 요약하면 다음과 같습니다.

1. Ollama로 EXAONE 3.5와 nomic-embed-text 모델 준비
2. Docker로 pgvector 데이터베이스 실행
3. 스키마 생성 → 시드 → 청킹 → 임베딩 순서로 데이터 구성
4. FastAPI 서버 실행 후 브라우저로 접속

> GPU가 있는 환경에서는 EXAONE 3.5 7.8B, GPU가 없는 환경에서는 2.4B 모델을 권장합니다.
